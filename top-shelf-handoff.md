# Top Shelf: Grocery Sort

> **⚠️ Current state lives in [`top-shelf-progress.md`](top-shelf-progress.md).** This file is the original design brief (still the source of truth for design *intent* and principles). The "Current prototype state" and "Open work" sections below are now out of date — the game has been built and extended well past them. Read the progress doc for what actually exists today.

A relaxing shelf-sorting puzzle game for iPhone (or web). Built as a gift — no ads, no in-app purchases, no lives system, ever. The whole point of this project is the *honest* version of a genre that's normally stuffed with monetization dark patterns.

A working HTML prototype exists (`index.html`, single file, no build step). This document describes the design, the prototype's current state, and what's left to do.

## Reference & origin

The mechanic is modeled on App Store games like *Goods Sorting: Sort Challenge* and similar "shelf sort" puzzles. **Mechanics are not copyrightable**, so cloning the play is legitimate; the project uses its own name, art direction, and theming. The name *Top Shelf: Grocery Sort* was chosen after checking the App Store — "Shelf Life" and "Goods Sort" were both crowded; "Top Shelf" had only one dormant non-game collision.

## Core mechanic

- The board is a set of **shelves**. Every shelf has exactly **3 slots**. This is standardized — no variable shelf sizes.
- Items have a type (e.g. 🍎, 🥕, 🍞). Three identical items on the same shelf = a match.
- A shelf **clears when it becomes monochrome** (all 3 slots hold the same type).
- The only verb is: **drag one item from one shelf to another shelf with room**. (Tap-to-select also supported as accessibility fallback.)
- Cleared shelves **stay in place as empty shelves** — they don't vanish or collapse. They become permanent breathing room.

That's the whole game loop. All challenge comes from routing items through scarce space.

## Difficulty levers

Three knobs, used in combination per level:

1. **Number of starting empty shelves.** More empty = easier. One empty = tight (near-deadlock pressure).
2. **Hidden back-row layers.** A slot can hold a stack: front item plus a greyed-out back item. Back items are *visible but identifiable* (you can see it's a red mug) — they just can't be moved until the front clears. This is the planning-uncertainty source: you can't fully pre-plan because you only learn back-row contents as you uncover them.
3. **Timer.** Generous timer on by default — soft pressure, not punishing. Players who want pure planning can toggle it off in **Easy mode** (global, not per-level).

Difficulty is **not** controlled by limited buffer slots — that mechanic doesn't fit this engine, because all items are present at the start and clearing only *frees* space. Difficulty also is not controlled by board shuffling on reset — boards are stable across retries so insight is rewarded.

## Fail states

- **Time up:** soft fail. Resets the same board. No lives lost, no penalty, no monetization hook.
- **Hard deadlock** (no legal move exists at all): ends the level immediately, shows a message explaining *why* ("No moves left — the board can't be solved from here"), and leaves the board visible so the player can see how they got there. This is a deliberate kindness — a deadlock is invisible to a casual player and feels like the game is broken if it just keeps ticking down a dead clock.
- **Soft deadlock** (legal moves exist but no winning path remains): not yet detected. See open work below.

## Monetization

None. No ads. No IAP. No coins, lives, hearts, energy, boosters, or any other resource that depletes and refills on a timer. If a feature requires inventing a depleting resource, it's the wrong feature.

## Anti-design (things we explicitly rejected)

These came up in design discussion and were ruled out. Documenting them so they don't get re-added:

- **Variable shelf sizes** — breaks the "monochrome = match" elegance.
- **Limited free spots / buffer slots** — doesn't fit a fully-visible board where space only grows over time.
- **Board shuffling on reset** — defeats the retry-and-learn loop that planning puzzles depend on. Anti-memorization should come from hidden layers, not relabeling positions.
- **Cleared shelves disappearing** — visually jarring, and removing them costs the player breathing room. Keep them as empty shelves.
- **Cleared shelves becoming locked/unusable** — considered as a difficulty-up option, rejected because it raises deadlock risk and cuts against the relaxing intent.
- **"Match 3" in the name** — wrong genre signal; this is a sort game, not a swap-grid match-3.

## Level generation

Levels are generated by **shuffling backward from a solved state**: start with each filled shelf monochrome, then perform random *legal reverse-moves* to scramble. By construction, the resulting board is solvable (the inverse sequence wins). Currently 6 hand-tuned level definitions in `LEVELS[]`, each specifying `kinds` (distinct grocery types), `emptyShelves` (starting breathing room), `layers` (whether back-row hidden items are allowed), `time`, and a blurb.

The 6 levels escalate roughly: 3 types / 2 empty / no layers → 6 types / 1 empty / with layers.

## Current prototype state

`index.html` is a single self-contained file. No build, no dependencies beyond two Google Fonts. Open in any browser.

Implemented:
- Core mechanic, drag-and-drop (pointer events, works for touch + mouse) and tap-select fallback.
- Greyed back-row hidden items.
- Generous timer with Easy-mode (∞) toggle.
- Unlimited retries, stable boards on reset.
- Hard-deadlock detection with an explanatory end card; board stays visible.
- 6 escalating levels.
- Cozy grocery aesthetic — warm cream/wood palette, Fraunces display + Nunito body fonts, emoji as placeholder grocery art.

## Open work (in rough priority order)

1. **Soft-deadlock detection.** Currently only hard deadlocks (no legal move at all) are caught. Soft deadlocks (legal moves exist but no win is reachable) would require a small solver — likely a bounded BFS/DFS over board states with memoization. Would let the game end softly and explain the situation instead of running out the clock on an unwinnable board. Watch for state-space blowup with 6+ types and back-row layers; may need depth limit or heuristic pruning.

2. **Conveyor-belt level type.** Originally part of the inspiration but not yet built. A real-time mode where items feed in over time onto a belt; player sorts them onto shelves before the belt overflows. Same match-rule, real-time pressure layered on top. This is a meaningfully different sub-engine (timed input, fail-on-overflow) and should be additive — a level type, not a replacement for the core puzzle mode.

3. **Real grocery art.** Currently emoji. Custom illustrated tiles (cans, boxes, produce, bottles) would give the game its own visual identity. Stay clear of any clone of *Goods Sort*'s specific art style — we're using our own theme deliberately.

4. **More levels.** 6 is enough for a prototype but not enough for the relaxing-pickup-and-play loop the design is aimed at. Aim for 30+ with carefully tuned escalation. Consider whether to also support an endless/daily-generated mode (only after soft-deadlock detection lands, since procedural levels need that safety net).

5. **Drag feel tuning on real touch devices.** The drag threshold, snap-back, and hover-highlight timing need play-testing on actual iPhone hardware, not just desktop browser. Currently uses default pointer-event thresholds; may need a small movement deadband to prevent accidental drags from finger-taps.

6. **Persistence.** No save state right now. At minimum: remember current level, Easy-mode preference, and how many levels are unlocked. `localStorage` is sufficient.

7. **iOS packaging.** If this ever leaves "web link sent to one person" territory, the path is either (a) PWA install (add-to-home-screen, works today as-is with a manifest + icon), or (b) a real native wrapper (Capacitor / Cordova) for App Store distribution — which would require an Apple Developer account and revisiting the name-availability check.

## Design principles to preserve

Three things hold the project together. Don't lose any of them in subsequent edits:

1. **The game must remain genuinely relaxing.** Generous timers, unlimited retries, gentle escalation. If a feature would make a casual player anxious, it's wrong for this game.
2. **No depleting resources, ever.** The moment the game has lives or hearts or coins, it has a paywall-shaped hole in it, even if nothing fills the hole. The architecture itself should make monetization impossible.
3. **Clarity over cleverness in the core mechanic.** 3-slot shelves, monochrome-clears, drag-to-move. Resist the urge to add powers, boosters, special tiles, combos. Variety comes from level configurations of the existing mechanic, not from new mechanics.
